# 단계별 작업 규칙

## 1. 문서 목적
이 프로젝트는 처음부터 모든 것을 동시에 만들지 않는다.
단계별로 구현하고, 각 단계마다 확인 기준을 통과한 뒤 다음 단계로 넘어간다.

---

## 2. 공통 원칙

### 2-1. 기능보다 흐름
한 단계에서는 핵심 흐름이 동작하는지를 우선 본다.

### 2-2. 완성 기준 없이 다음 단계로 넘어가지 않는다
각 단계는 “끝났다고 말할 수 있는 기준”이 있어야 한다.

### 2-3. 병목 후보는 문서에 남긴다
당장 해결하지 않아도,
문제가 될 수 있는 지점을 `04-problem-scenarios.md`에 연결한다.

---

## 3. Phase 1 - 기본 세팅과 문서화

목표:
- 프로젝트 구조 확정
- 문서 뼈대 완성
- MySQL 연결
- Flyway 기초 세팅
- Git 초기화

완료 기준:
- 프로젝트가 실행된다
- 로컬 DB 연결이 된다
- 문서 파일이 모두 존재한다
- 첫 커밋이 완료된다

---

## 4. Phase 2 - 인증과 종목 도메인

목표:
- 회원가입 / 로그인
- 종목 목록 / 상세 조회
- 기본 시세 데이터 구조

완료 기준:
- 인증 기본 흐름이 동작한다
- 종목 API가 동작한다
- API 문서가 최신 상태다

---

## 5. Phase 3 - 거래와 보유 종목

목표:
- 매수 / 매도
- holdings 반영
- 거래 내역 조회
- 보유 종목 조회

완료 기준:
- 거래 성공/실패 조건이 명확하다
- holdings와 trade_orders가 정합하게 반영된다
- 대표 조회 API가 동작한다

추가 규칙:
- 거래 관련 변경은 항상 정합성 검토를 포함한다

---

## 6. Phase 4 - 시세 생성기와 SSE

목표:
- 가짜 시세 생성
- 종목 현재가 갱신
- SSE 스트림 전송

완료 기준:
- 시세가 주기적으로 바뀐다
- 클라이언트가 SSE로 값을 받는다
- 기본 메트릭 후보가 정리된다

---

## 7. Phase 5 - 종목별 채팅과 WebSocket

목표:
- 채팅방 입장
- 메시지 전송/저장
- 실시간 브로드캐스트

완료 기준:
- 같은 방 사용자에게 메시지가 전송된다
- DB에 메시지가 저장된다
- chat_rooms.last_message 관련 컬럼이 정상 반영된다

---

## 8. Phase 6 - 읽기 성능 문제 재현

목표:
- 느린 초기 구현을 일부러 둔다
- 데이터량을 늘려 문제를 재현한다

대상:
- 채팅방 목록
- 거래 내역
- 메시지 목록
- 보유 종목 조회

완료 기준:
- 문제를 재현할 수 있다
- 측정 기준이 정해져 있다
- before 상태가 문서화되어 있다

---

## 9. Phase 7 - 성능 개선

목표:
- keyset pagination
- projection query
- 인덱스 설계
- unread count 개선
- N+1 완화

완료 기준:
- 개선 전후 비교가 있다
- 최소 하나 이상의 대표 API에서 수치 개선이 보인다
- 문서가 갱신된다

---

## 10. Phase 8 - 모니터링과 부하 테스트

목표:
- Actuator
- Prometheus
- Grafana
- k6 시나리오

완료 기준:
- 기본 대시보드가 동작한다
- 최소 2개 이상의 부하 테스트 시나리오가 있다
- 실시간 연결 수 또는 API latency를 수치로 확인할 수 있다

---

## 11. Phase 9 - 포트폴리오 정리

목표:
- README 정리
- 핵심 문제 해결 사례 정리
- before / after 수치 정리
- 기술면접 설명 문장 정리

완료 기준:
- 프로젝트 소개 문장이 명확하다
- 핵심 병목 해결 사례가 정리돼 있다
- 면접에서 3분 안에 설명 가능한 스토리가 있다

---

## 12. Codex 작업 규칙
Codex에게 작업을 시킬 때는 아래를 같이 요구한다.

1. 현재 작업이 어느 phase인지 먼저 확인
2. 해당 phase 범위를 넘지 않도록 구현
3. 끝나면 완료 기준 충족 여부 보고
4. 관련 문서 업데이트
5. 가능하면 커밋까지 수행

예시 지시문:
Read docs/12-phase-rules.md first.
Work only within the current phase scope.
After completing the task, update relevant docs and create a commit following docs/11-commit-rules.md.
