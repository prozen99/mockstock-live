# AI Context for Future Sessions

## 이 문서의 목적
새 채팅 세션에서 AI가 MockStockLive 프로젝트를 빠르게 이해하고,
이전 작업 방식과 품질 기준을 최대한 유지하도록 하기 위한 문서다.

---

## 프로젝트 성격
이 프로젝트는 단순 CRUD 프로젝트가 아니다.

목표는 다음을 보여주는 것이다.

- 정합성
- 성능 병목 인식
- 실시간 통신
- 문제 재현
- 측정
- 개선
- 문서화

즉 “기능이 된다”보다
“왜 문제가 생겼고, 무엇을 바꿨고, 어떤 수치가 좋아졌는지”
를 보여주는 프로젝트다.

---

## 작업 원칙
1. Phase 범위를 넘지 않는다.
2. 기능 추가보다 문제 재현과 개선 사례를 우선한다.
3. 코드만 바꾸고 끝내지 않는다.
4. before / after 측정과 문서화를 반드시 남긴다.
5. 남은 한계도 정직하게 기록한다.
6. local secret config 파일은 절대 수정/커밋하지 않는다.
7. 과한 리팩터링보다 명확하고 측정 가능한 개선을 선호한다.

---

## 문서화 원칙
새 작업은 반드시 아래를 남겨야 한다.

- 어떤 문제였는가
- 왜 중요한가
- 어떻게 재현했는가
- 무엇을 측정했는가
- 원인이 무엇이었는가
- 무엇을 바꿨는가
- 무엇이 얼마나 좋아졌는가
- 아직 남은 한계가 무엇인가

핵심 문서:
- `docs/13-troubleshooting.md`
- `docs/14-performance-lab.md`
- `docs/15-case-study-template.md`

---

## 현재까지 검증된 기능
- 회원가입 / 로그인
- 종목 조회
- 매수 / 매도 / 보유 종목 / 거래 내역
- 실시간 시세(SSE)
- 종목별 실시간 채팅(WebSocket/STOMP)
- 채팅 메시지 DB 저장
- 채팅방 목록 / 메시지 조회
- Phase 6 read path 개선 문서화

---

## 앞으로 AI가 조심해야 할 것
- 기능만 추가하고 문서를 안 남기지 않기
- “최적화했다”는 말만 하고 측정 수치를 안 남기지 않기
- unrelated file까지 불필요하게 건드리지 않기
- Phase와 상관없는 대규모 리팩터링 하지 않기
- 분산 시스템/대형 인프라(예: Kafka, Redis, 멀티 인스턴스)를 너무 이르게 도입하지 않기

---

## 새 Phase에서 우선적으로 해야 할 일
1. 기존 병목 후보 중 하나를 고른다
2. baseline을 만든다
3. 측정한다
4. 최소한의 개선을 적용한다
5. 다시 측정한다
6. 문서화한다
7. 면접용 요약까지 남긴다
